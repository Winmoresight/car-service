## 🎯 เข้าใจปัญหาก่อน

### ทำไมมี 1,051 รายการ แต่จริงๆ มีแค่ ~140 รายการที่เปลี่ยนยอดเงิน?

ระบบเก่าบันทึก **ทุกการแก้ไข** แม้ว่าจะเป็น:
- ✅ การเพิ่มชื่อธนาคาร (ไม่กระทบยอดเงิน) - ~900 รายการ
- ✅ การแก้ไขยอดเงินจริง (กระทบยอดเงิน) - ~140 รายการ
- ⚠️ การเข้าดูแล้วไม่ได้แก้ไข - บางรายการ

### ข้อมูลที่ถูก "โกง" หลายล้านมาจากไหน?

มาจาก **ผลรวมส่วนต่างของรายการที่แก้ไขยอดเงิน (~140 รายการ)**  
ไม่ใช่รวม 1,051 รายการทั้งหมด!

### ใช้ Filter เพื่อวิเคราะห์:

#### 1. **แสดงเฉพาะรายการที่แก้ไขยอดเงิน**
- เลือก Filter: **"🔴 แก้ไขยอดเงินเท่านั้น"**
- จะเห็นเฉพาะ ~140 รายการที่มีการเปลี่ยนยอดเงินจริง
- ดูที่คอลัมน์ **"ส่วนต่าง"** เพื่อดูว่าเงินเพิ่มหรือลด

#### 2. **แสดงเฉพาะรายการที่แก้ไขธนาคาร**
- เลือก Filter: **"🏦 แก้ไขธนาคารเท่านั้น"**
- จะเห็น ~900 รายการที่แก้ไขเฉพาะชื่อธนาคาร
- รายการเหล่านี้ไม่กระทบยอดเงิน (ปกติ)

#### 3. **ดู KPI Cards ด้านบน**
```
┌─────────────────────────┬─────────────────────────┬─────────────────────────┐
│  แก้ไขทั้งหมด           │  แก้ไขจำนวนเงินจริง       │  แก้ไขเฉพาะธนาคาร        │
│  1,051 ครั้ง            │  140 ครั้ง               │  900 ครั้ง               │
└─────────────────────────┴─────────────────────────┴─────────────────────────┘
```

#### 4. **ดูกล่องวิเคราะห์สีเหลือง**
- แสดงเปอร์เซ็นต์ของแต่ละประเภท
- แสดงผลรวมส่วนต่างเงินทั้งหมด
- มีสรุปชัดเจน

## 📊 การตีความผลลัพธ์

### ✅ สถานการณ์ปกติ (ไม่น่ากังวล)

#### 1. พนักงานแก้ไขเฉพาะธนาคารบ่อย + ส่วนต่าง ฿0
```
ต้นเตย: 820 รายการ, แก้ไขยอด 45, แก้ไขธนาคาร 770
```
**ความหมาย:** ต้นเตยมีหน้าที่เพิ่มชื่อธนาคารให้ลูกค้า (เป็นหน้าที่ปกติ)

#### 2. การแก้ไขยอดเงินเล็กน้อย (< ฿500)
```
บิล PSC626003948
- เงินสด: ฿740 → ฿750 (+฿10)
- เหตุผล: ลูกค้าจ่ายทิปเพิ่ม
```
**ความหมาย:** ปกติ - การปรับยอดเล็กน้อย

---

### 🟠 สถานการณ์น่าสงสัย (ควรตรวจสอบ)

#### 1. แก้ไขบิลเดียวกันหลายครั้งในเวลาสั้น
```
บิล PSC626003950 - แก้ไข 5 ครั้ง ใน 1 วัน
09:00 - ต้นเตย: ฿1,000 → ฿1,500
10:30 - พลอย: ฿1,500 → ฿1,200
11:00 - ต้นเตย: ฿1,200 → ฿1,000
14:00 - Admin: ฿1,000 → ฿2,000
16:00 - ต้นเตย: ฿2,000 → ฿500
```
**สัญญาณ:** อาจมีการปกปิดหรือแก้ไขข้อมูลเพื่อปิดบังการโกง

#### 2. แก้ไขยอดเงินนานหลังบิลปิด
```
บิล PSC626003900 - ออกวันที่ 1 มิ.ย.
แก้ไขวันที่ 5 มิ.ย. (4 วันหลังออกบิล)
- เงินสด: ฿5,000 → ฿2,000 (-฿3,000)
```
**สัญญาณ:** ทำไมต้องแก้ไขหลายวันหลังออกบิล?

---

### 🔴 สถานการณ์ร้ายแรง (โกงแน่นอน)

#### 1. ลดยอดเงินมากๆ โดยไม่มีเหตุผล
```
บิล PSC626004521
- เงินสด: ฿50,000 → ฿25,000 (-฿25,000)
- ผู้แก้ไข: ต้นเตย
- วันที่: 3 มิ.ย. 2026 14:25:33
```
**สัญญาณ:** เงิน ฿25,000 หายไป! ลูกค้าจ่ายจริงหรือไม่?

#### 2. พนักงานคนเดียวแก้ไขส่วนต่างสูง
```
ต้นเตย: แก้ไข 45 รายการ, ส่วนต่างรวม ฿1,245,000
พลอย: แก้ไข 40 รายการ, ส่วนต่างรวม ฿850,000
```
**สัญญาณ:** ทำไมต้นเตยแก้ไขส่วนต่างมากกว่าคนอื่นขนาดนี้?

#### 3. แก้ไขยอดเงินเป็นจำนวนกลมๆ
```
฿50,000 → ฿0
฿30,000 → ฿0
฿100,000 → ฿0
```
**สัญญาณ:** อาจเป็นการลบยอดเงินทั้งหมดเพื่อเอาเงินไปเก็บ

## 🚨 สัญญาณเตือนภัย (Red Flags)

### ระดับ 🔴 อันตราย (ต้องตรวจสอบทันที)

1. **ส่วนต่างเงิน > ฿10,000 ต่อรายการ**
2. **แก้ไขบิลเดียวกัน > 3 ครั้งใน 1 วัน**
3. **ลดยอดเงินเป็น ฿0 โดยไม่มีเหตุผล**
4. **แก้ไขบิลนานหลังออกบิล (> 7 วัน)**
5. **พนักงานคนเดียวมีส่วนต่างรวม > ฿500,000**

### ระดับ 🟠 น่าสงสัย (ควรติดตาม)

1. **ส่วนต่างเงิน ฿5,000 - ฿10,000**
2. **แก้ไขบิลเดียวกัน 2-3 ครั้ง**
3. **แก้ไขยอดเงินในเวลากลางคืน (หลัง 22:00)**
4. **แก้ไขบิล 3-7 วันหลังออกบิล**

### ระดับ 🟢 ปกติ (ไม่น่ากังวล)

1. **แก้ไขเฉพาะชื่อธนาคาร (ส่วนต่าง ฿0)**
2. **ส่วนต่างเงิน < ฿500**
3. **แก้ไขในวันเดียวกับออกบิล**

## 🎯 การดำเนินการต่อ

### ขั้นตอนที่ 1: รวบรวมหลักฐาน

```bash
# รัน script เพื่อเก็บผลลัพธ์
node analyze-edit-logic.js > fraud-analysis-report-$(date +%Y%m%d).txt
```

### ขั้นตอนที่ 2: ตรวจสอบบิลที่น่าสงสัย

ใช้ Frontend filter เพื่อดูรายละเอียด:
1. เลือก "🔴 แก้ไขยอดเงินเท่านั้น"
2. เรียงตาม "ส่วนต่าง" จากมากไปน้อย
3. บันทึกเลขบิลที่น่าสงสัย

### ขั้นตอนที่ 3: ตรวจสอบกับข้อมูลจริง

สำหรับแต่ละบิลที่น่าสงสัย:
- ดึงใบเสร็จจริงจากระบบเก่า
- ตรวจสอบกับยอดเงินที่ลูกค้าจ่ายจริง
- สอบถามพนักงานที่แก้ไข
- ตรวจสอบกล้อง CCTV (ถ้ามี)

### ขั้นตอนที่ 4: สร้าง Query เพิ่มเติม

```sql
-- หาบิลที่ถูกแก้ไขบ่อยที่สุด
SELECT 
  NumberPrint,
  COUNT(*) as edit_count,
  STRING_AGG(NameUser, ', ') as users,
  SUM(ABS((PayCashNew + PayTransferNew) - (PayCashOld + PayTransferOld))) as total_diff
FROM ChangeEditPrint
GROUP BY NumberPrint
HAVING COUNT(*) > 3
ORDER BY total_diff DESC;
```

### ขั้นตอนที่ 5: ดำเนินการตามความเหมาะสม

- **ถ้าพบการโกงจริง:** สอบสวนต่อ, ลงโทษตามระเบียบ
- **ถ้าไม่แน่ใจ:** สอบถามพนักงาน, ขอคำอธิบาย
- **ถ้าเป็นปกติ:** เก็บเป็นข้อมูลอ้างอิง

## 🛡️ การป้องกันไม่ให้เกิดซ้ำ

### 1. เพิ่มระบบ Approval
```
การแก้ไขยอดเงิน > ฿5,000 ต้อง:
- ผู้จัดการอนุมัติ
- ระบุเหตุผล
- แนบหลักฐาน (ภาพถ่าย, เอกสาร)
```

### 2. จำกัดสิทธิ์
```
- พนักงานทั่วไป: แก้ไขได้แค่ธนาคาร
- แคชเชียร์: แก้ไขยอดเงิน < ฿1,000
- ผู้จัดการ: แก้ไขยอดเงินได้ทั้งหมด
- Admin: ดูประวัติการแก้ไขทั้งหมด
```

### 3. เพิ่ม Alert แบบ Real-time
```
เมื่อมีการแก้ไขยอดเงิน > ฿5,000:
- ส่ง Email ไปยังผู้จัดการ
- ส่ง SMS Alert
- แสดง Notification ในระบบ
```

### 4. Lock บิลหลังเวลา
```
บิลที่เก่ากว่า 24 ชั่วโมง:
- ห้ามแก้ไข (lock)
- ถ้าจำเป็นต้องแก้ไข ต้องขออนุมัติพิเศษ
```
