# แผนเพิ่มฟอร์มบันทึกบิลคู่ค้า

## เป้าหมาย

เพิ่มความสามารถในหน้า `บิลคู่ค้า` ให้พนักงานบันทึกบิลซื้อ/บิลคู่ค้าที่มีเอกสารจริงอยู่แล้ว เพื่อใช้ติดตามยอดค้างชำระและรายการสินค้า โดยยังไม่บังคับให้เป็นระบบรับสินค้าเข้าสต็อกเต็มรูปแบบในรอบแรก

## สิ่งที่ตรวจพบจากระบบเดิม

- หน้าเว็บปัจจุบันอยู่ที่ `src/app/supplier-bills/page.tsx`
  - แสดงรายการบิลคู่ค้าแล้ว
  - แก้ไขได้เฉพาะสถานะและยอดเงินของบิลเดิม
  - ยังไม่มีหน้าสร้างบิลใหม่หรือเพิ่มรายการสินค้า
- API ปัจจุบันอยู่ที่ `src/app/api/supplier-bills/route.ts`
  - `GET /api/supplier-bills` อ่านหัวบิลและรายละเอียดบิล
  - `PATCH /api/supplier-bills` แก้สถานะและยอดเงิน
  - ยังไม่มี `POST` สำหรับสร้างบิลใหม่
- ตารางคู่ค้าที่ควรใช้คือ `dbo.Creditor`
  - `Company` ตอนนี้ไม่มีข้อมูล
  - `Creditor.Code` เป็น identity
  - `Creditor.CodeCompany` เป็นรหัสคู่ค้าแบบ 6 หลัก เช่น `000001`
  - คู่ค้าล่าสุดที่ตรวจพบคือ `000011`; ถ้าสร้างใหม่ควรรันเป็น `000012`
- ประเภทคู่ค้าดึงจาก `dbo.CaseCreditor`
  - ตัวอย่าง: ยาง, น้ำมันเครื่อง, อะไหล่, ประกัน, พรบ, ขนส่ง, ตรอ
- หัวบิลคู่ค้าอยู่ที่ `dbo.MasterPrintOderBuyProduct`
  - คอลัมน์หลัก: `DatePost`, `NumberPrintPost`, `CodeCompany`, `NameCompany`, `ReducePrice`, `Result`, `Reduceproduct`, `TotalPrice`, `Status`, `NameUser`, `CheckIn`
- รายการสินค้าในบิลอยู่ที่ `dbo.DetailPrintOderBuyProduct`
  - คอลัมน์หลัก: `DatePost`, `NumberPrintPost`, `AddRows`, `AddOder`, `BarCode`, `NameProduct`, `NumProduct`, `MeterProduct`, `SalePrice`, `RealSalePrice`, `ReducePrice`, `SumPrice`, `CostPrice`, `CodeCompany`, `CaseProduct`, `Status`, `CheckIn`
  - หลายคอลัมน์จำนวนเงิน/จำนวนใน detail เป็น `nvarchar` ไม่ใช่ `money` จึงต้อง format เป็นข้อความแบบเดิม
- เลขเอกสารคู่ค้าอยู่ที่ `dbo.NumberPrintOderBuyProduct`
  - ค่าปัจจุบันที่ตรวจพบ: `YearToday = 2026`, `ShortCH = POB`, `NumReport = 2`
  - ตัวอย่างเลขเดิม: `POB226000001`, `POB326000002`
  - สูตรที่น่าจะใช้: `POB` + เดือนแบบไม่เติมศูนย์ + ปี 2 หลัก + เลขรัน 6 หลัก
  - ตัวอย่างถัดไปในเดือน 6 ปี 2026 น่าจะเป็น `POB626000003`

## แนวทาง UX ที่เสนอ

เพิ่มปุ่ม `เพิ่มบิลคู่ค้า` บนหน้า `บิลคู่ค้า`

ใน dialog หรือหน้าใหม่ควรมีส่วนหลัก:

1. ข้อมูลคู่ค้า
   - ค้นหา/เลือกคู่ค้าจาก `Creditor`
   - แสดงรหัส, ชื่อ, ที่อยู่, โทร, เลขภาษี, ประเภทคู่ค้า
   - มีปุ่ม `เพิ่มคู่ค้าใหม่`
   - ถ้าเพิ่มคู่ค้าใหม่ ให้กรอกชื่อ, ประเภท, ที่อยู่, โทร, เลขภาษี, หมายเหตุ

2. ข้อมูลบิล
   - วันที่บิล default เป็นวันนี้
   - เลขเอกสารระบบ auto-generate แต่ควรแสดงให้เห็นก่อนบันทึก
   - สถานะ default เป็น `ค้างชำระ`
   - ผู้บันทึก default เป็น `admin` หรือชื่อผู้ใช้ที่ระบบมีในอนาคต
   - ช่องหมายเหตุ เก็บในรอบแรกฝั่ง UI ได้ก่อน แต่ถ้าจะเก็บจริงต้องยืนยันตาราง/คอลัมน์ เพราะหัวบิลเดิมไม่มีคอลัมน์หมายเหตุ

3. รายการสินค้า
   - เพิ่มแถวสินค้าได้หลายรายการ
   - ช่องที่ควรมีใน MVP:
     - รหัสสินค้า/บาร์โค้ด: optional
     - ชื่อสินค้า: required
     - จำนวน: required, มากกว่า 0
     - หน่วย: ดึงจาก `MeterProduct` หรือให้พิมพ์ได้ถ้าไม่พบ
     - ราคาต่อหน่วย: required
     - ส่วนลดต่อรายการ: optional, default 0
     - จำนวนเงิน: auto = จำนวน x ราคาต่อหน่วย - ส่วนลด
   - ไม่ควรบังคับผูกกับสินค้า master ในรอบแรก เพราะ user บอกว่าพนักงานมีบิลอยู่แล้วและกรอกเพื่อ tracking

4. สรุปยอด
   - ยอดรวมสินค้า = รวมจำนวนเงินทุกแถวก่อนส่วนลดท้ายบิล
   - ส่วนลดสินค้า = รวมส่วนลดในรายการ
   - ส่วนลดพิเศษ/ท้ายบิล = ช่อง optional
   - ยอดรวมสุทธิ = ยอดรวมสินค้า - ส่วนลดสินค้า - ส่วนลดพิเศษ
   - ค่า `TotalPrice` และ `Result` ในหัวบิลควรเก็บเป็นยอดสุทธิเดียวกันใน MVP

## แนวทาง API ที่เสนอ

เพิ่ม endpoint ภายใต้ `src/app/api/supplier-bills/route.ts`

### `GET /api/supplier-bills/catalog`

หรือสร้าง route ใหม่ เช่น `src/app/api/supplier-bills/catalog/route.ts`

ใช้สำหรับหน้า form:

- ดึงคู่ค้า (`Creditor`)
- ดึงประเภทคู่ค้า (`CaseCreditor`)
- ดึงหน่วย (`MeterProduct`)
- อาจดึงสินค้าเดิมจาก `MasterProduct` + `MasterProductDetail` เพื่อช่วย autocomplete ในอนาคต

### `POST /api/supplier-bills`

ใช้สร้างบิลใหม่แบบ transaction เดียว:

1. validate payload
2. ถ้าสร้างคู่ค้าใหม่:
   - lock/check `Creditor`
   - สร้าง `CodeCompany` ถัดไปแบบ 6 หลัก
   - insert `Creditor`
3. lock `NumberPrintOderBuyProduct`
4. สร้างเลขเอกสาร `NumberPrintPost`
5. insert `MasterPrintOderBuyProduct`
6. insert `DetailPrintOderBuyProduct` ทุกแถว
7. update `NumberPrintOderBuyProduct.NumReport`
8. commit transaction

ควรใช้ transaction จาก `mssql` โดยตรง เหมือนแนวทางใน API เปิดบิล เพื่อป้องกันเลขเอกสารชนกัน

## การเขียนข้อมูลลงตาราง

### `MasterPrintOderBuyProduct`

- `DatePost`: วันที่บิล
- `NumberPrintPost`: เลขเอกสารที่ generate
- `CodeCompany`: รหัสคู่ค้า
- `NameCompany`: ชื่อคู่ค้า
- `ReducePrice`: ส่วนลดพิเศษท้ายบิล
- `Result`: ยอดสุทธิ
- `Reduceproduct`: รวมส่วนลดสินค้า
- `TotalPrice`: ยอดสุทธิ
- `Status`: default `ค้างชำระ`
- `NameUser`: `admin` หรือผู้ใช้ปัจจุบัน
- `CheckIn`: `N`

### `DetailPrintOderBuyProduct`

- `DatePost`: วันที่บิล
- `NumberPrintPost`: เลขเอกสารเดียวกับหัวบิล
- `AddRows`: ลำดับ 1, 2, 3...
- `AddOder`: ลำดับแบบข้อความ
- `BarCode`: รหัส/บาร์โค้ด ถ้าไม่มีให้เป็นค่าว่าง
- `NameProduct`: ชื่อสินค้า
- `NumProduct`: จำนวน เป็นข้อความ
- `MeterProduct`: หน่วย
- `SalePrice`: ราคาต่อหน่วย เป็นข้อความทศนิยม 2 ตำแหน่ง
- `RealSalePrice`: ราคาต่อหน่วยก่อนส่วนลด หรือเท่ากับ `SalePrice` ใน MVP
- `ReducePrice`: ส่วนลดต่อรายการ เป็นข้อความทศนิยม 2 ตำแหน่ง
- `SumPrice`: จำนวนเงินสุทธิของแถว เป็นข้อความทศนิยม 2 ตำแหน่ง
- `CostPrice`: ควรเป็น `0.00` ใน MVP เว้นแต่ต้องการใช้เป็นราคาทุนแยกต่างหาก
- `CodeCompany`: รหัสคู่ค้า
- `CaseProduct`: ถ้าเลือกจากสินค้าเดิมให้ใส่ประเภทสินค้าเดิม, ถ้าเป็นรายการพิมพ์เองให้ default เป็น `25` หรือให้เลือกภายหลัง
- `Status`: สถานะเดียวกับหัวบิล
- `CheckIn`: `N`

## สิ่งที่ไม่ควรทำใน MVP

- ยังไม่ควรเพิ่ม stock เข้า `INOUTStockProduct` อัตโนมัติ
  - เหตุผล: user ระบุว่าเป็นการกรอกบิลที่มีอยู่แล้วเพื่อ tracking
  - ถ้าทำ stock เข้าเลย อาจทำให้สต็อกเพิ่มซ้ำกับระบบเดิมหรือเพิ่มก่อนรับของจริง
- ยังไม่ควรสร้างสินค้า master ใหม่จากรายการในบิลอัตโนมัติ
  - ถ้ารายการนั้นเป็นสินค้าจริงที่ต้องขายต่อ ควรทำ flow แยกหรือมี checkbox `สร้างเป็นสินค้าใหม่`
- ยังไม่ควรทำระบบพิมพ์ใบสั่งซื้อในรอบแรก
  - โฟกัสที่การบันทึกและติดตามยอดค้างชำระก่อน

## คำถามที่ควรยืนยันก่อนลงมือ

1. เลขเอกสารควรใช้สูตร `POB + เดือน + ปี 2 หลัก + เลขรัน 6 หลัก` ใช่ไหม
2. ช่องหมายเหตุของบิลต้องเก็บลงฐานข้อมูลจริงไหม ถ้าต้องเก็บ ควรเก็บที่ตารางไหนหรือเพิ่มตารางใหม่ในฝั่งเว็บ
3. รายการสินค้าที่กรอกในบิลคู่ค้าต้องกระทบสต็อกทันทีไหม หรือเก็บเพื่อ tracking ยอดจ่ายอย่างเดียว
4. ถ้าพนักงานพิมพ์สินค้าที่ไม่มีใน master ต้องให้สร้างสินค้าใหม่ทันทีไหม หรือเป็นรายการเฉพาะบิลได้
5. ผู้บันทึกใช้ `admin` ไปก่อน หรือมีแหล่งชื่อพนักงานในระบบที่ต้องผูกจริง

## ขั้นตอนดำเนินการเมื่อยืนยันแล้ว

1. เพิ่ม API catalog สำหรับคู่ค้า, ประเภทคู่ค้า, หน่วย และ autocomplete สินค้า
2. เพิ่ม `POST /api/supplier-bills` พร้อม transaction และเลขรัน
3. เพิ่มปุ่มและ form สร้างบิลคู่ค้าใน `src/app/supplier-bills/page.tsx`
4. เพิ่ม dialog สร้างคู่ค้าใหม่จากใน form
5. หลังบันทึกสำเร็จ refresh รายการบิลและเปิดดูรายละเอียดบิลที่สร้าง
6. ตรวจด้วย Biome เฉพาะไฟล์ที่แก้ตาม workflow ของโปรเจกต์
